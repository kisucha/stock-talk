| Kiwoom GetCommData 빈값 반환 | 2026-06-15 | pykiwoom이 `rqname`으로 명명한 두 번째 파라미터가 실제론 `sRecordName`임. 대부분 TR의 record name은 `""` (빈 문자열). rqname을 넣으면 항상 빈값. | `OnReceiveTrData` 콜백의 `record` 파라미터를 GetCommData에 직접 전달. 콜백 내부에서 즉시 데이터 추출. |

| pykrx 배치 API KRX 로그인 필요 | 2026-06-16 | `get_market_ohlcv_by_ticker`(배치)는 KRX 로그인 필요. 로그인 없으면 빈 응답 → 데이터 0건. `get_market_ohlcv_by_date`(단건)는 로그인 불필요. | 배치 API 사용 전 KRX 계정 여부 확인. 계정 없으면 단건 API로 교체. 상세 → `pykrx_오답노트.md` |

| MariaDB collation 불일치 JOIN 실패 | 2026-06-21 | 마이그레이션 005 신규 테이블 `COLLATE=utf8mb4_unicode_ci`로 만듦. 기존 테이블(stock_info 등)은 `utf8mb4_general_ci`. JOIN ON 시 "Illegal mix of collations" 던짐. 관심종목 로드 무조건 실패. catch에 잡혀 silent → 사용자 UI 빈 상태로 보임. | (1) 신규 마이그레이션은 기존 테이블 collation 확인 후 일치시킬 것 — 프로젝트는 `utf8mb4_general_ci`가 표준. (2) JOIN 키 컬럼은 collation 동일성 필수. (3) IPC 에러 console.warn에 res.error 반드시 포함 — 침묵 실패 방지. 마이그레이션 009로 005 테이블 전부 general_ci 변환. |
| stock_info.name 100자 초과 InsertFail | 2026-06-21 | `stock_info.name VARCHAR(100)` — KR 종목 기준 설계. US 종목 Security Name (특히 ETF/펀드 풀네임)이 100자 초과 → `yfinance.fetch_master()`의 `executemany` INSERT 시 `pymysql DataError (1406) "Data too long for column 'name'"`로 전체 트랜잭션 실패. US 마스터 0건 적재 → TQQQ 검색 시 자동 등록도 동일 사유로 차단. | 신규 데이터 소스 도입 시 외부 필드 최대 길이 사전 확인. KR-only 기준 컬럼 폭은 US/글로벌 확장 시 재검토 필수. 마이그레이션 011로 `name VARCHAR(255)` 확장. init.sql + 001_initial.sql도 동기화. |
| 브릿지↔렌더러 키 네이밍 불일치 | 2026-06-22 | Python 브릿지가 SSE 페이로드를 camelCase(`qty`, `execQty`, `orderNo`)로 송신, JS 렌더러는 다른 키(`volume`, `exec_qty`, `order_no`)로 참조 → `undefined` → 화면 빈 컬럼/오작동. 양쪽 모두 try-catch 안에 있어 silent fail. 호가 잔량(qty vs volume), 체결 통보 4개 필드(execQty/execPrice/orderNo/isFilled) 모두 동일 패턴. | (1) 신규 SSE 이벤트 타입 추가 시 송신측·수신측 키 1:1 매핑 표 작성 후 구현. (2) quote/orderbook/execution 등 같은 도메인 이벤트는 키 네이밍 컨벤션 통일(snake_case 또는 camelCase 일관). 본 프로젝트는 quote의 `volume`/`change` 등 단어형 컬럼을 기준으로 통일. (3) 렌더러 측에서 핵심 필드 누락 시 `console.warn` 출력 — silent `-` 표시 방지. (4) 근본 원인 발견 시 동일 이벤트 채널의 다른 페이로드도 함께 검사. |
| 키움 실시간 fid 매핑 오류 | 2026-06-22 | 호가 페이로드에서 잔량 fid를 `45+i`, `55+i`로 사용 → 실제로는 매도호가 6~10, 매수호가 6~10. 매도잔량 정확 fid는 61~65, 매수잔량은 71~75. 결과: 잔량 컬럼에 가격 데이터 표시. SetRealReg fid_str에 잔량 fid 누락 — 일부 환경에서 잔량 미수신 가능성. | (1) 키움 실시간 fid 표 정확히 참조: 매도호가1~5=41~45 / 매도호가6~10=46~50 / 매수호가1~5=51~55 / 매수호가6~10=56~60 / 매도잔량1~5=61~65 / 매도잔량6~10=66~70 / 매수잔량1~5=71~75 / 매수잔량6~10=76~80. (2) `GetCommRealData(code, fid)` 호출 시 fid 번호와 의미를 주석으로 명시. (3) `SetRealReg` fid_str에 실제 사용 fid 모두 포함 — 시세+매도호가+매수호가+매도잔량+매수잔량 묶어 등록. |
| 체결내역 메모리 캐시 휘발 | 2026-06-22 | 렌더러 `state.pendingOrders`/`state.execHistory`를 메모리 push로 누적 → 창 닫고 재오픈 시 빈 상태. 키움 OpenAPI에 이미 체결/미체결 데이터 있는데도 활용 안 함. | 실시간 거래 관련 모든 데이터(체결/미체결/계좌/잔고/보유종목)는 메모리 캐시 대신 키움 OpenAPI TR 호출(OPT10075=미체결+체결 통합, OPW00004=계좌평가잔고, OPW00001=예수금 등) 기준. 메모리 state는 응답 덮어쓰기 캐시 용도로만 사용. 관심종목만 DB 영속화 허용.
