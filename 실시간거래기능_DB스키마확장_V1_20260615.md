## 1. 현황 분석

### 1.1 기존 아키텍처 (Phase 1~3)

**현재 특성**:
- DB: MariaDB (192.168.20.80:3306) 중앙 집중식
- 데이터: 사용자 수동 CSV import 방식 (실시간 X)
- 거래: 수동 분석 전용 (실시간 체결 기능 X)
- 계좌: 메모리 기반 `user_holdings` 수동 관리

**문제점**:
- 실시간 시세 연동 불가 (Phase 4 FinanceDataReader 이전까지)
- 거래 이력 추적 불가
- 계좌 잔고 자동 동기화 불가
- KIS 포지션과 DB 데이터 간 불일치 위험

### 1.2 목표 아키텍처 (Phase 6 예상)

**추가 요소**:
1. **KIS API 연동**: 인증, 주문 실행, 체결 통보, 계좌 조회
2. **실시간 시세 수신**: WebSocket (H0STCNI0 지수, H0STCNI1 개별종목)
3. **거래 이력 추적**: trading_orders 테이블
4. **계좌 스냅샷**: trading_account 테이블 (일별)
5. **토큰 캐싱**: kis_credentials 테이블 (24시간 유효)

## 2. KIS API 개요

### 2.1 KIS Open API 특성

**공급사**: 한국투자증권 공식 API
**기본 정보**:
- **Base URL**: `https://openapi.koreainvestment.com:9443` (상용), `https://openapi.koreainvestment.com:9443` (모의)
- **인증 방식**: AppKey + AppSecret + AccessToken (JWT, 유효기간 24시간)
- **토큰 갱신**: 매일 자동 갱신 권장 (토큰 만료 1시간 전)

**핵심 API**:

| API 명 | 메서드 | 경로 | 용도 |
|--------|--------|------|------|
| 인증 | POST | `/oauth2/tokenP` | AccessToken 발급 |
| 주식주문 | POST | `/uapi/domestic-stock/v1/trading/order-cash` | 매수/매도 주문 |
| 주문상태 | GET | `/uapi/domestic-stock/v1/trading/inquire-daily-orders` | 주문 이력 조회 |
| 잔고 조회 | GET | `/uapi/domestic-stock/v1/trading/inquire-balance` | 보유 종목 + 잔고 |
| 손익 조회 | GET | `/uapi/domestic-stock/v1/trading/inquire-daily-ccld` | 당일 체결 현황 |
| 실시간 시세 | WebSocket | `/tryitout` | H0STCNI0/H0STCNI1 체결가/거래량 |
| 체결 통보 | WebSocket | - | H0STCNI0 (체결 실시간) |

### 2.2 주문 요청 구조

**요청 본문** (매수 예시):

```json
{
  "CANO": "12345678",        // 계좌번호 앞 8자리
  "ACNT_PRDT_CD": "01",      // 계좌번호 뒷 2자리
  "PDNO": "053800",          // 종목 코드
  "ORD_QTY": "100",          // 주문 수량
  "ORD_UNPR": "54000",       // 주문 가격
  "BUY_SELL_GB": "01",       // 매수(01) / 매도(02)
  "ORD_DVSN_CD": "00"        // 시장가(01) / 지정가(00)
}
```

**응답 본문**:

```json
{
  "rt_cd": "0",              // 성공(0) / 실패(1)
  "msg_cd": "01000000",      // 상세 코드
  "msg": "성공",
  "output": {
    "KRX_FWDG_ORD_NO": "00000123456", // KIS 주문번호
    "ODNO": "1",                       // 브로커 주문번호
    "ORD_TMD": "093000"                // 주문 시간 (HHmmss)
  }
}
```

### 2.3 실시간 체결 통보 (WebSocket H0STCNI0)

**메시지 예시** (체결 발생 시):

```json
{
  "TRD_TIME": "153000",      // 거래 시간 (HHmmss)
  "STCK_PRPR": "54200",      // 현재가
  "ACML_VOL": "5000000",     // 누적 거래량
  "STCK_OPRC": "54100",      // 시가
  "STCK_HGPR": "54500",      // 고가
  "STCK_LWPR": "53900"       // 저가
}
```

### 2.4 계좌 잔고 응답 구조

**주요 필드**:

```json
{
  "output1": {
    "isum_total_amt": "1000000",    // 총 자산
    "nass_amt": "500000",            // 예수금
    "tass_amt": "1500000",           // 총 자산 (평가금)
    "pcch_amt": "50000"              // 당일 손익
  },
  "output2": [
    {
      "pdno": "053800",
      "hldg_qty": "100",            // 보유 수량
      "prpr": "54200",              // 현재가
      "pchs_amt": "5420000",        // 매입금액
      "evlu_amt": "5420000",        // 평가금액
      "evlu_pfls_amt": "0"          // 평가 손익
    }
  ]
}
```

## 3. 신규 테이블 설계

### 3.1 kis_credentials 테이블

**목적**: KIS API 인증 정보 캐싱 + 토큰 갱신 관리

**컬럼 설계**:

| 컬럼명 | 타입 | 길이 | NULL | 키 | 설명 |
|--------|------|------|------|-------|------|
| id | INT | - | NO | PK | 자동증가 |
| account_no | VARCHAR | 16 | NO | UQ | 계좌번호 (예: 12345678-01) |
| app_key | VARCHAR | 256 | NO | - | KIS App Key (암호화) |
| app_secret | VARCHAR | 256 | NO | - | KIS App Secret (암호화) |
| access_token | TEXT | - | YES | - | JWT AccessToken (암호화) |
| token_issued_at | DATETIME | - | YES | - | 토큰 발급 시간 |
| token_expires_at | DATETIME | - | YES | - | 토큰 만료 시간 (발급 후 24시간) |
| approval_key | VARCHAR | 256 | NO | - | API 승인요청 키 (모의투자 용) |
| is_paper | TINYINT(1) | - | NO | - | 모의투자(1) / 실투(0) |
| tr_id | VARCHAR | 20 | NO | - | API TrId (고정: "CSPAT00001" 등) |
| created_at | TIMESTAMP | - | NO | - | 생성 시간 |
| updated_at | TIMESTAMP | - | NO | DEF | 마지막 갱신 |

**암호화 고려사항**:
- app_secret, access_token, app_key: Node.js `crypto` 모듈로 AES-256 암호화
- 마스터 키: .env에 `ENCRYPT_KEY` 저장 (별도 보관)

**인덱스**:
```
PRIMARY KEY (id)
UNIQUE KEY uq_account (account_no)
INDEX idx_token_expires (token_expires_at)
```

### 3.2 trading_orders 테이블

**목적**: 주문 이력 추적, 수익률 계산, 거래 검증

**컬럼 설계**:

| 컬럼명 | 타입 | 길이 | NULL | 키 | 설명 |
|--------|------|------|------|-------|------|
| id | INT | - | NO | PK | 자동증가 |
| account_no | VARCHAR | 16 | NO | FK | kis_credentials 참조 |
| ticker | VARCHAR | 20 | NO | FK | stock_info 참조 |
| order_type | ENUM | - | NO | - | '매수'/'매도' |
| order_qty | INT | - | NO | - | 주문 수량 |
| order_price | INT | - | NO | - | 주문 가격 (지정가) |
| kis_order_no | VARCHAR | 20 | YES | UQ | KIS 주문번호 (고유) |
| status | ENUM | - | NO | - | 접수/미체결/부분체결/체결/취소 |
| exec_qty | INT | - | YES | - | 체결 수량 |
| exec_price | INT | - | YES | - | 평균 체결가 |
| exec_amount | BIGINT | - | YES | - | 체결금액 (= exec_qty × exec_price) |
| commission | INT | - | YES | - | 수수료 |
| tax | INT | - | YES | - | 세금 (매도 시) |
| is_paper | TINYINT(1) | - | NO | - | 모의투자(1) / 실투(0) |
| notes | TEXT | - | YES | - | 사용자 메모 (진입 근거 등) |
| created_at | TIMESTAMP | - | NO | - | 주문 시간 |
| executed_at | TIMESTAMP | - | YES | - | 체결 시간 |
| updated_at | TIMESTAMP | - | NO | DEF | 마지막 업데이트 |

**계산 필드** (앱 레벨):
- `손익 = (exec_qty × exec_price) - (주문금액 + 수수료 + 세금)`
- `손익률 = 손익 / 주문금액 × 100`

**인덱스**:
```
PRIMARY KEY (id)
UNIQUE KEY uq_kis_order (kis_order_no)
INDEX idx_account_ticker (account_no, ticker)
INDEX idx_status_created (status, created_at)
FOREIGN KEY (account_no) REFERENCES kis_credentials(account_no)
FOREIGN KEY (ticker) REFERENCES stock_info(ticker)
```

### 3.3 trading_account 테이블

**목적**: 일별 계좌 스냅샷, 수익률 차트, 성과 분석

**컬럼 설계**:

| 컬럼명 | 타입 | 길이 | NULL | 키 | 설명 |
|--------|------|------|------|-------|------|
| id | INT | - | NO | PK | 자동증가 |
| account_no | VARCHAR | 16 | NO | FK | kis_credentials 참조 |
| deposit | BIGINT | - | NO | - | 예수금 |
| eval_total | BIGINT | - | NO | - | 평가 자산 (주식 평가가) |
| eval_stock | BIGINT | - | YES | - | 주식 평가금액 |
| cash | BIGINT | - | YES | - | 현금 잔고 |
| pnl_today | BIGINT | - | YES | - | 당일 손익 |
| pnl_total | BIGINT | - | YES | - | 누적 손익 |
| rate_of_return | DECIMAL(8,2) | - | YES | - | 수익률 (%) |
| is_paper | TINYINT(1) | - | NO | - | 모의투자(1) / 실투(0) |
| snapshot_at | DATE | - | NO | UQ(복합) | 스냅샷 날짜 |
| created_at | TIMESTAMP | - | NO | - | 저장 시간 |

**계산 방식**:
- `eval_total = 주식 평가금액 + 현금`
- `pnl_total = eval_total - deposit`
- `rate_of_return = pnl_total / deposit × 100`

**인덱스**:
```
PRIMARY KEY (id)
UNIQUE KEY uq_account_date (account_no, snapshot_at)
INDEX idx_snapshot_date (snapshot_at)
FOREIGN KEY (account_no) REFERENCES kis_credentials(account_no)
```

### 3.4 realtime_watchlist 테이블

**목적**: 실시간 모니터링 종목 목록, WebSocket 구독 관리

**컬럼 설계**:

| 컬럼명 | 타입 | 길이 | NULL | 키 | 설명 |
|--------|------|------|------|-------|------|
| id | INT | - | NO | PK | 자동증가 |
| ticker | VARCHAR | 20 | NO | FK | stock_info 참조 |

| alert_price_high | INT | - | YES | - | 상단 알람가 (선택) |
| alert_price_low | INT | - | YES | - | 하단 알람가 (선택) |
| alert_volume_ratio | DECIMAL(4,2) | - | YES | - | 거래량 배수 알람 (예: 2.0배) |
| is_active | TINYINT(1) | - | NO | - | 활성화(1) / 비활성화(0) |

**인덱스**:

## 4. user_holdings 확장 전략

### 4.1 현재 구조 (Phase 1~3)

```sql
CREATE TABLE user_holdings (
  id              INT AUTO_INCREMENT PRIMARY KEY,
  ticker          VARCHAR(20)  NOT NULL,
  avg_price       INT,                          -- 평균 매입가
  quantity        INT,                          -- 보유 수량
  available_cash  INT,                          -- 가용 자금
  strategy        TEXT,                         -- 투자 전략
  horizon         TEXT,                         -- 투자 기간
  expected_issue  TEXT,                         -- 기대 이슈
  split_plan      JSON,                         -- 분할 매수 계획
  updated_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  FOREIGN KEY (ticker) REFERENCES stock_info(ticker),
  UNIQUE KEY uq_ticker (ticker)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**문제점**:
- KIS 실시간 데이터와의 동기화 방법 명확하지 않음
- 평가손익 컬럼 없음 (렌더러에서 계산)
- 모의투자 vs 실투 구분 불가

### 4.2 확장 스키마 제안 (Phase 6)

**선택지 1: 기존 테이블 컬럼 추가** (마이그레이션)

```sql
ALTER TABLE user_holdings ADD COLUMN (
  kis_sync_at         DATETIME,                 -- KIS 마지막 동기화 시간
  eval_price          INT,                      -- 현재 평가가 (KIS에서 실시간 수신)
  eval_amount         BIGINT,                   -- 평가금액 (qty × eval_price)
  pnl_amount          BIGINT,                   -- 평가손익 (eval_amount - 매입금액)
  pnl_rate            DECIMAL(8,2),             -- 손익률 (%)
  is_paper            TINYINT(1) DEFAULT 0,     -- 모의투자(1) / 실투(0)
  kis_hldg_qty        INT,                      -- KIS 조회 보유수량 (검증용)
  kis_avg_price       INT                       -- KIS 조회 평균가 (검증용)
);

CREATE INDEX idx_eval_price ON user_holdings(eval_price);
CREATE INDEX idx_is_paper ON user_holdings(is_paper);
```

**선택지 2: 별도 실시간 테이블** (권장 — 기존 코드 영향 최소화)

```sql
CREATE TABLE user_holdings_realtime (
  id                INT AUTO_INCREMENT PRIMARY KEY,
  ticker            VARCHAR(20) NOT NULL,
  eval_price        INT,                       -- 현재 평가가
  eval_amount       BIGINT,
  pnl_amount        BIGINT,
  pnl_rate          DECIMAL(8,2),
  kis_sync_at       DATETIME,
  is_paper          TINYINT(1),
  updated_at        TIMESTAMP,
  UNIQUE KEY uq_ticker (ticker),
  FOREIGN KEY (ticker) REFERENCES stock_info(ticker)
);
```

**추천: 선택지 2**
- 기존 user_holdings는 수동 입력용으로 유지
- user_holdings_realtime은 KIS API 자동 동기화용
- 렌더러에서 두 테이블 조인하여 표시

### 4.3 동기화 전략

**매 시간 (또는 30분 주기)**:

```sql
UPDATE user_holdings_realtime
SET eval_price = (SELECT prpr FROM kis_balance WHERE ticker = ?),
    eval_amount = quantity × eval_price,
    pnl_amount = eval_amount - (avg_price × quantity),
    pnl_rate = (pnl_amount / (avg_price × quantity)) × 100,
    kis_sync_at = NOW()
WHERE ticker = ? AND is_paper = ?
```

**실시간 WebSocket 수신 시** (H0STCNI1 개별종목):
- 메모리 캐시 (Map) 즉시 업데이트
- DB는 1분~5분 주기로 배치 갱신

## 5. 데이터 흐름 파이프라인

### 5.1 흐름 1: 실시간 시세 수신

**구성요소**:
1. KIS WebSocket 클라이언트 (main.js)
2. 메모리 캐시 (Map<ticker, {price, change, volume, timestamp}>)
3. IPC 채널 푸시 (renderer→main)
4. DOM 업데이트 (렌더러)

**상세 흐름**:

```
┌─ main.js (Electron 메인)
│  └─ WebSocket 클라이언트
│     └─ KIS: H0STCNI1 체결가 수신 (실시간)
│        ├─ 메모리 Map 즉시 업데이트
│        ├─ IPC: "rt:priceUpdate" 푸시 → 렌더러
│        └─ 1분마다 DB에 배치 저장 (trading_account 스냅샷)
│
└─ 렌더러 (index.html)
   └─ IPC 수신 → Chart 리드로우
```

**메모리 캐시 구조**:

```javascript
// main.js
const realtimePrices = new Map(); // {ticker: {price, change, volume, timestamp, high, low}}
const subscriptions = new Set(); // 구독 중인 종목 리스트

// 수신 예
realtimePrices.set('053800', {
  price: 54200,
  change: 200,
  volume: 1000000,
  high: 54500,
  low: 53900,
  timestamp: '153000'
});
```

**IPC 채널 추가**:

| 채널명 | 방향 | 데이터 | 용도 |
|--------|------|--------|------|
| `rt:priceUpdate` | main → renderer | {ticker, price, change, volume, timestamp} | 실시간 시세 푸시 |

## 8. 보안 및 토큰 관리

### 8.1 암호화 전략

**저장되는 민감 정보**:
- app_key, app_secret, access_token

**암호화 방식**: AES-256-CBC (Node.js crypto)

**구현**:

```javascript
// utils/encryption.js
const crypto = require('crypto');

const ENCRYPT_KEY = process.env.ENCRYPT_KEY; // .env에서 로드
const ENCRYPT_IV = Buffer.from([...32바이트...]); // 고정 또는 DB 저장

function encrypt(text) {
  const cipher = crypto.createCipheriv('aes-256-cbc', ENCRYPT_KEY, ENCRYPT_IV);
  let encrypted = cipher.update(text, 'utf8', 'hex');
  encrypted += cipher.final('hex');
  return encrypted;
}

function decrypt(encrypted) {
  const decipher = crypto.createDecipheriv('aes-256-cbc', ENCRYPT_KEY, ENCRYPT_IV);
  let decrypted = decipher.update(encrypted, 'hex', 'utf8');
  decrypted += decipher.final('utf8');
  return decrypted;
}

module.exports = { encrypt, decrypt };
```

### 8.2 토큰 관리 흐름

**초기 발급**:

```
1. 사용자 KIS 계정 정보 입력 (main.js 설정창)
2. KIS API POST /oauth2/tokenP 호출
3. AccessToken 반환 (유효 24시간)
4. kis_credentials 테이블 INSERT (암호화 저장)
5. 메모리 tokenCache에 로드
```

**자동 갱신**:

```
토큰 만료 1시간 전 (매 1시간마다 체크)
  ↓
토큰 갱신 API 호출 (동일 요청)
  ↓
새 토큰 반환
  ↓
kis_credentials.access_token 업데이트
  ↓
메모리 캐시 갱신
```

## 12. KIS API 에러 처리

### 12.1 주요 에러 코드

| 코드 | 의미 | 처리 |
|------|------|------|
| 01000000 | 성공 | - |
| 10000001 | 인증 실패 | 토큰 갱신 후 재시도 |
| 10000002 | 토큰 만료 | 토큰 갱신 |
| 10000003 | 주문 한도 초과 | 사용자 알림 |
| 10000004 | 계좌 오류 | 계좌 정보 재확인 |
| 20000000 | 시스템 오류 | 재시도 로직 |

### 12.2 에러 처리 패턴

```javascript
// aiService.js 또는 별도 kis-service.js
async function placeOrder(account, ticker, qty, price, type, isPaper) {
  try {
    // 1. 토큰 유효성 확인
    if (tokenCache.expires_at < Date.now()) {
      await refreshToken();
    }

    // 2. API 호출
    const response = await fetch(`${KIS_BASE_URL}/uapi/domestic-stock/v1/trading/order-cash`, {
      method: 'POST',
      headers: {
        'Authorization': `Bearer ${tokenCache.access_token}`,
        'Content-Type': 'application/json'
      },
      body: JSON.stringify({...})
    });

    const data = await response.json();

    // 3. 응답 검증
    if (data.rt_cd !== '0') {
      throw new Error(`KIS API 에러: ${data.msg_cd} - ${data.msg}`);
    }

    // 4. DB 저장
    await db.query(
      'INSERT INTO trading_orders (kis_order_no, status, ...) VALUES (?, ?, ...)',
      [data.output.KRX_FWDG_ORD_NO, '접수', ...]
    );

    // 5. IPC 응답
    return {success: true, kis_order_no: data.output.KRX_FWDG_ORD_NO};
    
  } catch (error) {
    logger.error('KIS_API', 'Order failed', error);
    
    // 6. 에러 분류
    let userMessage = '주문 실패';
    if (error.message.includes('10000002')) {
      userMessage = '인증 만료 — 다시 로그인하세요';
    } else if (error.message.includes('10000003')) {
      userMessage = '주문 한도 초과';
    }
    
    return {success: false, error: userMessage};
  }
}
```
