## 연구 항목 1: Chart.js + chartjs-chart-financial 플러그인

### 1.1 chartjs-chart-financial 플러그인 개요

Chart.js는 기본적으로 기본 차트 유형(선형, 막대, 원형 등)만 지원하며, 금융 차트의 핵심인 **캔들스틱(Candlestick) 차트와 OHLC(Open-High-Low-Close) 차트는 기본 내장되지 않는다**. 이를 해결하기 위해 `chartjs-chart-financial` 플러그인이 필수적이다.

**플러그인의 핵심 기능:**
- 캔들스틱(OHLC) 차트 렌더링
- OHLC 바 차트 렌더링
- 금융 데이터의 X축 시간 스케일 처리
- 대량의 OHLC 데이터 효율적 렌더링

### 1.2 설치 및 기본 구조

npm을 통한 설치:
```
npm install chart.js chartjs-chart-financial
```

플러그인의 핵심 구조는 Chart.js의 Register 시스템을 활용한다. 플러그인 로드 후, Chart 객체에 금융 차트 유형(`candlestick`, `ohlc`)을 사용할 수 있게 된다.

**플러그인 동작 원리:**
1. Chart.js 코어 엔진이 데이터 배열을 처리 시 플러그인이 그려야 할 도형의 좌표를 계산
2. 각 데이터 포인트의 X 위치 계산 (스케일 적용)
3. OHLC 값을 기반으로 캔들 또는 바 형태의 도형 생성
4. 도형을 Canvas에 렌더링

### 1.3 캔들스틱 차트 데이터 형식

chartjs-chart-financial에서 요구하는 **정확한 데이터 형식**:

```javascript
// 기본 형식
{
  x: "2026-06-11",  // 또는 Date 객체 또는 숫자(밀리초 타임스탐프)
  o: 56000,         // Open (시가)
  h: 57200,         // High (고가)
  l: 55800,         // Low (저가)
  c: 56500          // Close (종가)
}
```

**X축 형식의 중요성:**
- 문자열("2026-06-11"): 가장 안정적, 시간 스케일 처리 자동화
- Date 객체: JavaScript 표준, 밀리초 단위 정확도 제공
- 숫자(타임스탐프): 연속적인 시간 흐름 표현 가능, 최고 성능

**데이터 배열 구조:**
```javascript
const candleData = [
  { x: "2026-06-11", o: 56000, h: 57200, l: 55800, c: 56500 },
  { x: "2026-06-10", o: 56100, h: 56800, l: 55900, c: 56000 },
  { x: "2026-06-09", o: 56300, h: 57000, l: 56100, c: 56200 }
];
```

### 1.4 멀티 Y축 설정 (3패널 구성)

설계 문서의 8장에서 요구하는 3패널 구조:

**패널 1 (메인)**: 캔들 + 볼린저밴드 + MA5/20/60 | 가격 축 (Y축)
**패널 2 (OBV)**: OBV + OBV MA20 | 거래량 축 (Y축) — 가격 축과 다른 스케일
**패널 3 (RSI)**: RSI(14) + 30/70 기준선 | 0~100 축 (Y축)

Chart.js에서 **멀티 Y축 구현 방법**:

각 데이터셋(Dataset)이 독립적인 Y축을 가질 수 있도록 설정한다. 예를 들어:

```javascript
const config = {
  type: 'candlestick',  // 또는 'line', 'bar'
  data: {
    labels: ['2026-06-11', '2026-06-10', ...],
    datasets: [
      {
        label: 'OHLC',
        data: candleData,
        yAxisID: 'y-price'  // 가격 축 지정
      },
      {
        label: 'MA5',
        type: 'line',
        data: ma5Data,
        yAxisID: 'y-price',  // 동일 가격 축
        borderColor: '#fff'
      },
      {
        label: 'OBV',
        type: 'line',
        data: obvData,
        yAxisID: 'y-obv'  // 독립 OBV 축
      },
      {
        label: 'RSI',
        type: 'line',
        data: rsiData,
        yAxisID: 'y-rsi'  // 독립 RSI 축
      }
    ]
  },
  options: {
    scales: {
      'y-price': {
        type: 'linear',
        position: 'left',
        title: { display: true, text: '가격(원)' }
      },
      'y-obv': {
        type: 'linear',
        position: 'right',
        title: { display: true, text: 'OBV' }
      },
      'y-rsi': {
        type: 'linear',
        position: 'right',
        min: 0,
        max: 100,
        title: { display: true, text: 'RSI(14)' }
      }
    }
  }
};
```

### 1.5 멀티 패널 구성: 3개 독립 Canvas vs 하나의 Chart

**중요한 설계 결정**: 3개의 독립 Canvas를 사용하는 것이 권장된다.

**이유:**
1. **메모리 효율**: 각 패널이 독립적 메모리 할당 → 렌더링 최적화
2. **스크롤/줌 동기화 복잡도 감소**: 각 패널의 X축 범위를 정확히 일치시켜야 함
3. **성능**: 단일 차트에서 3개 Y축 + 다양한 지표를 처리하면 렌더링 부하 증가
4. **반응형 레이아웃**: 각 패널을 독립적으로 resize 가능

**단일 Chart에서 멀티 패널을 구현할 경우의 한계:**
- Y축 공간이 3개 필요 (좌측 1개, 우측 2개) → 가독성 저하
- X축 레이블 공간 제한
- 각 패널 높이 비율 조정 복잡도 증가

**권장 구조:**
```html
<div class="chart-container">
  <div class="chart-panel-1">
    <canvas id="chartPriceVolume"></canvas>
  </div>
  <div class="chart-panel-2">
    <canvas id="chartOBV"></canvas>
  </div>
  <div class="chart-panel-3">
    <canvas id="chartRSI"></canvas>
  </div>
</div>
```

### 1.9 차트 반응형 레이아웃 (Electron 창 크기 변경 대응)

Electron 앱에서 창 크기 변경 시 차트가 자동으로 리사이징되어야 한다.

**구현 방법:**

1. **Canvas 부모 컨테이너에 relative positioning 설정:**
```css
.chart-container {
  position: relative;
  width: 100%;
  height: 400px;  /* 또는 calc() 사용 */
}

canvas {
  display: block;
  max-width: 100%;
}
```

2. **Chart.js 옵션에 반응형 설정:**
```javascript
options: {
  responsive: true,
  maintainAspectRatio: false,
  layout: {
    padding: {
      left: 50,
      right: 50,
      top: 10,
      bottom: 10
    }
  }
}
```

3. **Electron IPC를 통한 resize 이벤트 처리:**
```javascript
window.addEventListener('resize', () => {
  // 차트가 자동으로 리사이징되나?
  // Chart.js의 responsive 옵션으로 자동 처리되지만,
  // 필요시 명시적으로 redraw 가능
  chart.resize();
});
```

4. **Electron 메인 프로세스에서 창 이벤트 감지:**
Electron의 `win.on('resize', ...)` 이벤트를 감지하여
렌더러 프로세스에 notify하고 차트 재렌더링 트리거 가능.

## 연구 항목 2: 기술적 지표 계산 알고리즘

### 2.1 OBV (On-Balance Volume)

#### 2.1.1 기본 정의 및 계산식

OBV는 거래량의 누적값을 이용해 수급 방향을 판단하는 지표이다. 가격 상승일에는 거래량을 가산하고, 하락일에는 거래량을 차감하는 방식으로 누적한다.

**기본 계산식:**

```
OBV[t] = OBV[t-1] + (volume[t] × sign[t])

여기서:
  OBV[t]      = t번째 봉의 OBV
  OBV[t-1]    = (t-1)번째 봉의 OBV
  volume[t]   = t번째 봉의 거래량
  sign[t]     = { +1 if close[t] > close[t-1]
                { 0  if close[t] == close[t-1]
                { -1 if close[t] < close[t-1]
```

**초기값 설정:**
```
OBV[0] = volume[0]  // 첫 번째 봉의 OBV는 그 거래량
```

#### 2.1.2 계산 예시 (5개 봉)

| 일자 | 종가(원) | 거래량(주) | 전일대비 | OBV 계산 | OBV 누적 |
|------|---------|----------|--------|---------|--------|
| 2026-06-07 | 56,000 | 100,000 | - | 100,000 | 100,000 |
| 2026-06-08 | 56,500 | 120,000 | ↑ | +120,000 | 220,000 |
| 2026-06-09 | 56,200 | 90,000 | ↓ | -90,000 | 130,000 |
| 2026-06-10 | 56,100 | 110,000 | ↓ | -110,000 | 20,000 |
| 2026-06-11 | 56,800 | 150,000 | ↑ | +150,000 | 170,000 |

#### 2.1.3 OBV MA20 계산

OBV 자체는 거래량의 누적으로 인해 지수적으로 증가하는 경향이 있어, **20일 단순이동평균(SMA)**을 함께 사용하여 추세를 판단한다.

```
OBV MA20[t] = (OBV[t] + OBV[t-1] + ... + OBV[t-19]) / 20
```

**신호 판단:**
- `OBV > OBV MA20`: 최근 거래량 누적이 평균보다 높음 = 수급 유입 신호
- `OBV < OBV MA20`: 최근 거래량 누적이 평균보다 낮음 = 수급 이탈 신호
- `OBV == OBV MA20`: 거래량 추세 중립

#### 2.1.4 OBV 다이버전스 감지 알고리즘

**일반 불리시 다이버전스** (약세장에서의 반등 신호):

조건:
1. 가격의 저점(Low)이 계속 낮아지는 중
2. OBV의 저점(Local Bottom)은 계속 높아지는 중

의미: 가격은 계속 내려가지만 수급은 유입되고 있다 = 반등 가능성 증가

**감지 알고리즘 절차:**

```
Step 1: 최근 N개 봉(권장 10~20개)에서 가격의 Local Bottom 찾기
        → 비교: close[i-1] > close[i] < close[i+1]

Step 2: 같은 기간에서 OBV의 Local Bottom 찾기
        → 비교: obv[i-1] > obv[i] < obv[i+1]

Step 3: 가격 저점들의 시퀀스 분석
        Step 3a: 가장 최근 저점과 그 이전 저점 비교
                가격_최근저점 < 가격_이전저점 ? (가격이 더 내려감)

Step 4: OBV 저점들의 시퀀스 분석
        Step 4a: obv_최근저점 > obv_이전저점 ? (OBV는 더 올라감)

Step 5: 조건 확인
        if (가격_최근저점 < 가격_이전저점) AND (obv_최근저점 > obv_이전저점):
            → 일반 불리시 다이버전스 감지
```

**구현 시 주의사항:**

1. **국소 최저값(Local Minimum) 정의의 정확성**
   - 단순히 이전 값과 다음 값을 비교하면 노이즈에 민감
   - 권장: 3개 봉 이상의 window를 사용하여 추세 방향 확인

2. **다이버전스 신뢰도 강화**
   - 저점 사이의 시간 간격: 최소 5일 이상 권장 (너무 짧으면 노이즈)
   - 저점의 명확성: 저점 주변의 변동폭이 커야 신뢰도 증가

3. **False Signal 방지**
   - 단순 다이버전스만으로 진입하지 말 것
   - 추가 조건 확인: RSI < 40 (약세 지속), 거래량 회복, MA 교차 등

### 2.2 볼린저밴드 (Bollinger Bands)

#### 2.2.1 기본 정의 및 계산식

볼린저밴드는 20일 이동평균(중심선) 주변에 표준편차 2배를 상/하단으로 하는 변동성 지표이다.

**3개 선의 계산식:**

```
1. 중심선 (Middle Band):
   MB[t] = SMA20[t] = (close[t] + close[t-1] + ... + close[t-19]) / 20

2. 표준편차 계산:
   σ[t] = sqrt( Σ(close[t-i] - MB[t])² / N )
   
   여기서 N = 20 (모분산 사용)
   또는 N-1 = 19 (표본분산 사용 — pandas 기본값)

3. 상단선 (Upper Band):
   UB[t] = MB[t] + 2 × σ[t]

4. 하단선 (Lower Band):
   LB[t] = MB[t] - 2 × σ[t]
```

**모분산 vs 표본분산 선택:**
- **모분산 (N=20)**: 현재 20개 봉이 전체 모집단이라고 가정 → 더 좁은 밴드폭
- **표본분산 (N-1=19)**: 현재 20개 봉이 더 큰 모집단의 샘플이라고 가정 → 더 넓은 밴드폭

권장: **표본분산 (N-1)** 사용. 이유는 pandas, numpy, 대부분의 금융 소프트웨어가 기본값으로 사용.

#### 2.2.2 %B (Percent B) 계산

%B는 현재 가격이 볼린저밴드 내에서 어디에 위치하는지를 0~1 범위로 정규화한 값이다.

```
%B[t] = (close[t] - LB[t]) / (UB[t] - LB[t])

판단 기준:
  %B < 0  : 현재가가 하단선 아래 이탈 (극단적 약세)
  0 < %B < 0.2  : 하단선 근처 (진입 신호)
  0.2 < %B < 0.8  : 밴드 내부 (정상 상태)
  0.8 < %B < 1.0  : 상단선 근처 (청산 신호)
  %B > 1  : 현재가가 상단선 위 이탈 (극단적 강세)
```

#### 2.2.3 밴드폭 (Bandwidth) 계산

밴드폭은 상단선과 하단선 사이의 거리를 중심선으로 정규화한 값으로, 변동성의 크기를 측정한다.

```
Bandwidth[t] = (UB[t] - LB[t]) / MB[t]

또는 정규화 버전:
Bandwidth_norm[t] = (Bandwidth[t] - Bandwidth_min) / (Bandwidth_max - Bandwidth_min)

판단 기준:
  Bandwidth < 0.05  : BB 스퀴즈 (변동성 극히 낮음) → 큰 움직임 예상
  0.05 < Bandwidth < 0.10  : 변동성 낮음 (확장 가능성)
  0.10 < Bandwidth < 0.20  : 정상 변동성
  0.20 < Bandwidth < 0.30  : 높은 변동성
  Bandwidth > 0.30  : 극도로 높은 변동성 (추세 마감 신호)
```

#### 2.2.4 BB 스퀴즈 감지 기준

**스퀴즈(Squeeze)**: 밴드폭이 최근 20일 중 최저값 근처에 있는 상태

```
1. 최근 20일 밴드폭의 최소값 계산:
   BW_min_20d = min(Bandwidth[t], Bandwidth[t-1], ..., Bandwidth[t-19])

2. 스퀴즈 조건:
   if Bandwidth[t] < BW_min_20d × 1.1:  // 최저값의 10% 범위
       → BB 스퀴즈 상태

3. 스퀴즈 해제 신호:
   (Bandwidth[t] - Bandwidth[t-1]) > 0  (밴드폭 증가 추세)
   AND Volume[t] > Volume_MA20[t]  (거래량 증가)
   → 스퀴즈 후 폭발적 움직임 예상
```

### 2.3 RSI (Relative Strength Index)

#### 2.3.1 기본 정의 및 Wilder 평활 방식

RSI는 **Wilder의 평활 방식(Wilder's Smoothing)**을 사용하는 것이 표준이다. 이는 지수이동평균(EMA)과 유사하지만, 초기값 계산과 평활 계수가 다르다.

**계산 절차 (14일 RSI):**

```
Step 1: 일일 변화 계산
  ΔClose[t] = close[t] - close[t-1]

Step 2: 상승분과 하락분 분리
  Gain[t] = max(ΔClose[t], 0)  // 양수만 취함
  Loss[t] = max(-ΔClose[t], 0)  // 절대값

Step 3: 초기값 (첫 14일) 계산 — 단순평균
  평활_이전 = sum(Gain[1..14]) / 14
  평활_이전 = sum(Loss[1..14]) / 14

Step 4: Wilder 평활 적용 (15일부터)
  평활[t] = (평활[t-1] × 13 + Gain[t]) / 14
  평활[t] = (평활[t-1] × 13 + Loss[t]) / 14

Step 5: RS와 RSI 계산
  RS[t] = 평활_상승[t] / 평활_하락[t]
  RSI[t] = 100 - (100 / (1 + RS[t]))
         = 100 × 평활_상승[t] / (평활_상승[t] + 평활_하락[t])
```

#### 2.3.2 Wilder 평활과 EMA의 차이

| 특성 | Wilder 평활 (RSI 표준) | EMA |
|------|---------------------|-----|
| 평활계수 | 1/N | 2/(N+1) |
| 초기값 | 첫 N개 단순평균 | 첫 값 = close[0] |
| 이전값 가중치 | (N-1)/N | (N-1)/(N+1) |
| RSI 계산 | 상승/하락 별도 평활 | 직접 적용 불가 |

**예시 계산 (간단한 14일):**

| 일자 | 종가(원) | 변화 | Gain | Loss | 평활_상승 | 평활_하락 | RSI |
|------|---------|------|------|------|----------|----------|-----|
| Day 1 | 100 | - | - | - | - | - | - |
| Day 14 | 104 | +4 | ... | ... | (합계)/14 | (합계)/14 | 계산값 |
| Day 15 | 105 | +1 | +1 | 0 | (이전×13+1)/14 | (이전×13+0)/14 | 계산값 |
| Day 16 | 103 | -2 | 0 | 2 | (이전×13+0)/14 | (이전×13+2)/14 | 계산값 |

#### 2.3.3 RSI 다이버전스 감지 (고급)

**최근 N개 피봇 포인트 비교 알고리즘:**

```
1. 피봇 포인트 찾기 (최근 20개 봉에서):
   - 가격 피봇: 좌우 2개씩 비교하여 Local High/Low 찾기
   - RSI 피봇: 동일하게 Local High/Low 찾기

2. 최근 2개 저점(또는 고점) 비교:
   
   저점 다이버전스 감지:
   if (Price_Low[recent] < Price_Low[previous]) 
      AND (RSI_Low[recent] > RSI_Low[previous]):
       → 약세 시장에서의 반등 신호 (신뢰도 ★★★★)

   고점 다이버전스 감지:
   if (Price_High[recent] > Price_High[previous])
      AND (RSI_High[recent] < RSI_High[previous]):
       → 상승 시장에서의 하락 신호 (신뢰도 ★★★★)

3. 추가 신뢰도 강화 조건:
   - 피봇 포인트 사이의 시간 간격: 5일 이상
   - RSI 수준 확인: 저점 다이버전스 시 RSI < 40, 고점 다이버전스 시 RSI > 60
   - 거래량 확인: 다이버전스 발생 시 거래량 증가 확인
```

**히든 불리시 다이버전스** (고급 신호):

```
조건:
  - 가격: 저점이 점점 높아짐 (상승 추세)
  - RSI: 저점이 점점 낮아짐 (약화 신호)

의미: 가격은 상승하지만 모멘텀이 약화 → 조정 후 재상승 가능성
신뢰도: ★★★★★ (매우 높음)

감지 방법:
if (Price_Low[recent] > Price_Low[previous])
   AND (RSI_Low[recent] < RSI_Low[previous]):
    → 히든 불리시 다이버전스 (추세 지속 신호)
```

### 2.4 MACD (Moving Average Convergence Divergence)

#### 2.4.1 기본 정의 및 계산식

MACD는 두 개의 지수이동평균(EMA)의 차이를 이용하여 추세 전환을 포착하는 지표이다.

**3개 선의 계산식:**

```
1. EMA12 계산 (12일 지수이동평균):
   평활계수₁ = 2 / (12 + 1) = 2/13 ≈ 0.1538
   
   EMA12[t] = close[t] × 평활계수₁ + EMA12[t-1] × (1 - 평활계수₁)
   초기값: EMA12[0] = close[0] (또는 처음 12개의 SMA)

2. EMA26 계산 (26일 지수이동평균):
   평활계수₂ = 2 / (26 + 1) = 2/27 ≈ 0.0741
   
   EMA26[t] = close[t] × 평활계수₂ + EMA26[t-1] × (1 - 평활계수₂)

3. MACD 계산:
   MACD[t] = EMA12[t] - EMA26[t]

4. 시그널 계산 (MACD의 9일 EMA):
   평활계수₃ = 2 / (9 + 1) = 2/10 = 0.2
   
   Signal[t] = MACD[t] × 평활계수₃ + Signal[t-1] × (1 - 평활계수₃)

5. 히스토그램:
   Histogram[t] = MACD[t] - Signal[t]
```

#### 2.4.2 골든크로스/데드크로스 신호

**골든크로스** (매수 신호):
```
조건:
  MACD[t-1] < Signal[t-1]  AND  MACD[t] > Signal[t]
  또는: Histogram[t-1] < 0  AND  Histogram[t] > 0

의미: 단기 모멘텀이 장기 추세를 상향 돌파 → 상승 신호
```

**데드크로스** (매도 신호):
```
조건:
  MACD[t-1] > Signal[t-1]  AND  MACD[t] < Signal[t]
  또는: Histogram[t-1] > 0  AND  Histogram[t] < 0

의미: 단기 모멘텀이 장기 추세를 하향 돌파 → 하락 신호
```

#### 2.4.3 구현 시 주의사항

1. **초기값 계산의 정확성**
   - 첫 번째 EMA12: close[0]으로 설정하면 계산 오차 발생 가능
   - 권장: 처음 12개 봉의 SMA를 초기값으로 사용

2. **EMA 계산의 누적 효과**
   - EMA는 이전값에 의존하므로, 과거 데이터가 많을수록 정확
   - 권장: 최소 50개 봉 이상의 데이터에서 MACD 계산 시작

3. **히스토그램의 극값(Peak) 감지**
   - 히스토그램의 절대값이 최대에서 감소하는 시점 = 모멘텀 약화
   - 이는 추세 전환의 조기 신호가 될 수 있음

### 2.5 스토캐스틱 오실레이터 (Stochastic Oscillator)

#### 2.5.1 기본 정의 및 K, D 계산

스토캐스틱은 일정 기간 동안의 최고가와 최저가 범위 내에서 현재가의 상대적 위치를 측정한다.

**K 계산 (14일 기본):**

```
K[t] = ((close[t] - LOW[t-13...t]) / (HIGH[t-13...t] - LOW[t-13...t])) × 100

여기서:
  LOW[t-13...t]   = 최근 14일간의 최저가
  HIGH[t-13...t]  = 최근 14일간의 최고가

범위: 0 ~ 100
```

**D 계산 (K의 3일 단순이동평균):**

```
D[t] = SMA3(K) = (K[t] + K[t-1] + K[t-2]) / 3
```

#### 2.5.2 신호 판단

| K 값 범위 | 의미 |
|----------|------|
| 0 ~ 20 | 과매도 (매수 신호) |
| 20 ~ 80 | 정상 구간 |
| 80 ~ 100 | 과매수 (매도 신호) |

**K와 D의 교차:**
- K가 D를 상향 돌파: 단기 강세 신호
- K가 D를 하향 돌파: 단기 약세 신호

### 2.6 ATR (Average True Range)

#### 2.6.1 기본 정의 및 True Range 계산

ATR은 일일 변동폭(Volatility)을 측정하는 지표로, 손절과 목표가 설정에 주로 사용된다.

**True Range (TR) 계산:**

```
TR[t] = max(
  High[t] - Low[t],              // 당일 고저폭
  |High[t] - Close[t-1]|,        // 고가와 전일종가의 차이
  |Low[t] - Close[t-1]|          // 저가와 전일종가의 차이
)
```

**ATR 계산 (14일 Wilder 평균):**

```
Step 1: 초기 ATR (첫 14일)
  ATR_initial = (TR[1] + TR[2] + ... + TR[14]) / 14

Step 2: Wilder 평균 (15일부터)
  ATR[t] = (ATR[t-1] × 13 + TR[t]) / 14

또는 평활계수 형태:
  ATR[t] = ATR[t-1] × (1 - 1/14) + TR[t] × (1/14)
         = ATR[t-1] × 0.9286 + TR[t] × 0.0714
```

#### 2.6.2 손절 및 목표가 설정

```
손절 기준선:
  손절선 = 진입가 - (ATR × 1.5 ~ 2.0배)
  
  예: 진입가 54,500원, ATR = 500원
     보수적: 54,500 - (500 × 2) = 53,500원
     공격적: 54,500 - (500 × 1.5) = 53,750원

목표가 설정:
  목표가 = 진입가 + (ATR × 2.5 ~ 3.5배)
  
  예: 진입가 54,500원, ATR = 500원
     보수적: 54,500 + (500 × 2.5) = 55,750원
     공격적: 54,500 + (500 × 3.5) = 56,250원
```

### 2.7 CCI (Commodity Channel Index)

#### 2.7.1 기본 정의 및 계산식

CCI는 현재 가격이 일정 기간의 평균 가격에서 얼마나 벗어났는지를 측정하는 지표이다.

```
Step 1: 전형가격(Typical Price) 계산:
  TP[t] = (High[t] + Low[t] + Close[t]) / 3

Step 2: 전형가격 20일 SMA:
  TP_MA[t] = (TP[t] + TP[t-1] + ... + TP[t-19]) / 20

Step 3: 평균편차(MAD, Mean Absolute Deviation) 계산:
  MAD[t] = (|TP[t] - TP_MA[t]| + |TP[t-1] - TP_MA[t]| + ... + |TP[t-19] - TP_MA[t]|) / 20

Step 4: CCI 계산:
  CCI[t] = (TP[t] - TP_MA[t]) / (0.015 × MAD[t])

상수 0.015: 금융 시장 표준 (Commodity Channel Index의 원래 정의)
```

#### 2.7.2 신호 판단

| CCI 값 | 의미 |
|--------|------|
| CCI > 100 | 과매수 (매도 신호) |
| CCI > 0 | 상승 추세 |
| -100 < CCI < 100 | 횡보 (방향성 약함) |
| CCI < 0 | 하락 추세 |
| CCI < -100 | 과매도 (매수 신호) |

### 2.8 캔들 패턴 감지 알고리즘

#### 2.8.1 망치형 (Hammer)

**정의**: 아래꼬리가 길고, 위꼬리가 짧으며, 몸통이 작은 패턴.

**조건:**
```
몸통 = abs(close - open)
아래꼬리 = open 또는 close 중 낮은 값 - low  (음수이므로 절대값 취함)
위꼬리 = high - (open 또는 close 중 높은 값)

망치형 조건:
  1. 아래꼬리 > 몸통 × 2      (아래꼬리가 몸통 길이의 2배 이상)
  2. 위꼬리 < 몸통 × 0.5      (위꼬리가 몸통 길이의 절반 미만)
  3. close > open             (양봉, 또는 close ≈ high)
  4. 전일 캔들: 음봉 또는 긴 음봉 (하락 추세 후 출현)

신호: 하락 후 출현 시 강한 반등 신호 (신뢰도 ★★★★)
```

#### 2.8.2 역망치형 (Inverted Hammer)

**정의**: 위꼬리가 길고, 아래꼬리가 짧으며, 몸통이 작은 패턴.

**조건:**
```
역망치형 조건:
  1. 위꼬리 > 몸통 × 2        (위꼬리가 몸통 길이의 2배 이상)
  2. 아래꼬리 < 몸통 × 0.5    (아래꼬리가 몸통 길이의 절반 미만)
  3. close < open             (음봉, 또는 close ≈ low)
  4. 다음날 캔들: 양봉 확인 필수 (신호 확정)

신호: 하락 후 출현하고 다음날 양봉 시 반등 신호 (신뢰도 ★★★)
```

#### 2.8.3 도지 (Doji)

**정의**: 시가와 종가가 거의 같아서 몸통이 없는 패턴.

**조건:**
```
도지 조건:
  abs(close - open) < (high - low) × 0.1   (몸통이 전일 고저폭의 10% 미만)
  또는 abs(close - open) < 100원 (절대값, 종목별 조정 필요)

변형:
  - 긴 다리 도지: 위/아래 꼬리가 모두 긴 경우 (방향성 약함)
  - 스핀탑 도지: 몸통은 중앙, 꼬리가 짧은 경우 (추세 전환 신호)

신호: 추세 전환 신호, 매수/매도 결정 어려움 (신뢰도 ★★★)
```

#### 2.8.4 강세장악형 (Bullish Engulfing)

**정의**: 전일 음봉을 당일 양봉이 완전히 포함하는 패턴.

**조건:**
```
강세장악형 조건:
  1. 전일: 음봉
     open[t-1] > close[t-1]

  2. 당일: 양봉
     open[t] < close[t]

  3. 완전 포함:
     open[t] <= close[t-1]  (당일 시가가 전일 종가 이하)
     close[t] >= open[t-1]  (당일 종가가 전일 시가 이상)

신호: 강한 매수 전환 신호, 추세 반전 (신뢰도 ★★★★★)
```

#### 2.8.5 저점갭업 (Gap Up with Strong Close)

**정의**: 전일 종가보다 높게 시작하여 양봉으로 마감하는 패턴.

**조건:**
```
저점갭업 조건:
  1. 오늘 시가 > 어제 종가 (갭업 발생)
     open[t] > close[t-1]

  2. 오늘 양봉 (강한 매수)
     close[t] > open[t]

  3. 거래량 (선택, 신뢰도 강화)
     volume[t] > volume_ma20[t] × 1.2

신호: 강한 매수세 유입, 상승 신호 (신뢰도 ★★★★)
```

async function sendMessage() {
  const input = document.querySelector('.chat-input textarea');
  const message = input.value.trim();
  if (!message) return;

  // 사용자 메시지 표시
  addChatMessage(message, 'user');

  const responseElement = document.createElement('div');

    const response = await window.api.ai.streamChat(message);

      responseElement.textContent += chunk;
      // 자동 스크롤
      document.querySelector('.chat-history').scrollTop = 
        document.querySelector('.chat-history').scrollHeight;
    }
  } catch (error) {
    addChatMessage(`오류: ${error.message}`, 'assistant');

}

  document.querySelector('.chat-history').scrollTop =

### 4.2 지표 계산에 필요한 최소 데이터 포인트 수

|------|------------------|-----------------|--------|------|
| MA20 | 20 | - | 20 | 20일 이동평균 기본값 |

| MA120 | 120 | - | 120 | 중기 추세 판단 |

| OBV MA20 | 20 | 5~10 | 25~30 | OBV 추세선 |
| RSI(14) | 14 | 14 | 28 | Wilder 평활 2주기 필요 |
| MACD | 26 | 35 | 35 | EMA26 + 시그널 안정화 |
| 스토캐스틱 | 14 | 10 | 24 | K와 D 계산 |

- MACD가 가장 많은 데이터 필요 (EMA26 안정화)
- RSI, ATR 등도 Wilder 평활로 인해 2주기 필요
- 안정적인 지표 계산을 위해 초기값 계산 후 추가 20개 정도

### 4.3 페이지네이션 없이 전체 로드 vs 스크롤 기반 동적 로드

**전체 로드 방식 (권장: 이 프로젝트):**

```
장점:
1. 구현 단순함 (모든 데이터를 메모리에 올리고 차트 그리기)
2. 차트 상호작용 빠름 (줌, 팬 시 재로드 불필요)
3. 지표 계산 1회로 끝남

단점:
1. 메모리 사용량 증가 (10년 2500개 레코드 × 50KB ≈ 125MB)
2. 초기 로딩 시간 (2500개 지표 계산 ≈ 1~2초)

권장 사유: Electron 데스크탑 앱 (메모리 충분), 사용자 편의성
```

**스크롤 기반 동적 로드 (대안):**

```
구현 방식:
1. 초기 로드: 최근 100개 봉만 로드 (3~4개월)
2. 사용자가 왼쪽(과거)으로 스크롤 시:
   ├─ 현재 표시 범위의 첫 번째 날짜 감지
   ├─ DB에서 추가 100개 데이터 쿼리
   ├─ 기존 데이터에 병합
   └─ 차트 재렌더링

JavaScript 구현:
chart.options.plugins.pan = {

장점: 메모리 절약, 로딩 시간 단축
단점: 복잡한 구현, 지표 계산 일관성 (새 데이터 추가 시 일부 지표 재계산)
```

### 4.4 메모리 효율: 10년치 데이터(약 2500개 레코드) 처리

**메모리 계산:**

```
단일 OHLCV 레코드:
  date (문자열 10자) + o, h, l, c, v (각 4바이트 정수)
  = 10 + 4×5 = 30바이트 (대략)

2500개 레코드:
  2500 × 30 = 75KB (Raw OHLCV)

지표 데이터 추가:
  MA5, MA20, MA60, MA120 = 4 × 8바이트 (float64) = 32바이트
  BB 상단, 중심, 하단, %B = 4 × 8 = 32바이트
  OBV, OBV_MA20 = 2 × 8 = 16바이트
  RSI, MACD, Signal, Histogram = 4 × 8 = 32바이트
  스토캐스틱 K, D = 2 × 8 = 16바이트
  ATR, CCI = 2 × 8 = 16바이트

지표당 약 144바이트/레코드
2500 × 144 = 360KB

총 메모리:
  75KB (Raw) + 360KB (지표) = 435KB

실제 JavaScript 객체 오버헤드 (~3배):
  435KB × 3 = 1.3MB
```

**최적화 기법:**

1. **TypedArray 사용** (메모리 70% 절약):
```javascript
// 일반 배열 (비효율)
const ma20 = new Array(2500).fill(0);  // 각 요소에 객체 메타데이터

// TypedArray (효율)
const ma20 = new Float64Array(2500);  // 순수 숫자 배열, 메타데이터 없음
```

2. **필요한 지표만 계산**:
현재 화면에 표시되는 지표만 미리 계산, 나머지는 필요시 lazy-load

3. **Canvas 메모리 최적화**:
3개 패널의 Canvas를 병렬 렌더링하되, 필요시 비트맵 이미지로 캐시

**핵심 정리:**

4. **데이터 흐름**: DB → 지표 계산 → 차트 렌더링의 명확한 파이프라인 제시
